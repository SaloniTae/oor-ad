OOR-AD — API-FIRST PLATFORM (v1)  ·  OVERVIEW & INDEX
================================================================================
Base URL:   https://<your-host>/api/v1
Every endpoint (except GET /api/v1/ping) requires:   x-api-key: <your key>
================================================================================

WHAT THIS IS
    A complete, documented, secured API surface that mirrors every action in
    the website. A third party can build their OWN player UI from scratch using
    only these endpoints — zero dependency on the embed page.

    This surface (/api/v1/*) is fully separate from the website's own routes
    (/v1/*). It has its own auth (x-api-key), its own standard error envelope,
    and its own 404 handling.

AUTH IN ONE LINE
    Send  x-api-key: <key>  on every call. Admin/full-access keys can do
    everything; restricted keys carry explicit scopes (read-only by default).

STANDARD RESPONSES
    - Every response carries a request_id (body + x-request-id header).
    - Every error:
        { "error_code":"...", "message":"...", "field":"<if applicable>",
          "request_id":"req_..." }
    - Auth error codes: MISSING_KEY, INVALID_KEY, REVOKED, PAUSED, EXPIRED,
      TENANT_DISABLED, ADMIN_ONLY, INSUFFICIENT_SCOPE, RATE_LIMITED.
    - Common: VALIDATION_ERROR, NOT_FOUND, *_NOT_FOUND, INTERNAL_ERROR.

SCOPES
    *                  full access (admin)
    channels:read/write
    playback:write
    sessions:read/write
    ads:read/write
    telemetry:read

SECTION FILES
    01-keys.txt        Admin / API-key management (create, revoke, pause,
                       rotate, list, usage). ADMIN key required.
    02-channels.txt    Channel CRUD (m3u8|mp4|mkv|youtube|direct_url), portable
                       stream reference.
    03-playback.txt    Signed pin-based playback URLs, payload on/off toggle,
                       active sessions, revoke/kick.
    04-ads.txt         Register creatives, trigger/cancel ad breaks, real-time
                       ad-break state (bumper / current ad / pod progress).
    05-telemetry.txt   Concurrent viewers, session/ad/error events, device
                       breakdown. Real-time snapshot + historical query.

PUSH VS POLL
    - POLL: /ads/state, /telemetry/realtime, /playback/sessions.
    - PUSH: viewer WebSocket  wss://host/ws?channel=<slug>&token=<viewer jwt>
            (play_pod / resume_live / session_terminated) and account webhooks
            (ad.break_started, ad.break_canceled, ad.completed).

CONCURRENCY / SCALE
    Sessions, device limits, kicks, ad-break state, presence, and rate limits
    are all Redis-backed and fan out over Redis pub/sub, so behaviour is
    correct across multiple cluster workers under concurrent load.

QUICK START (typical third-party flow)
    1. Admin: POST /admin/keys                       -> get a key (shown once)
    2. POST /channels                                -> create a channel
    3. POST /playback/generate                       -> pin + playback_url + token
    4. Your player: POST /v1/stream/authorize {pin,deviceId,channelSlug}
                    -> streamToken -> load /v1/stream/manifest?stoken=...
    5. POST /ads/trigger                             -> start an ad break
    6. GET  /ads/state / /telemetry/realtime         -> render your own overlays
